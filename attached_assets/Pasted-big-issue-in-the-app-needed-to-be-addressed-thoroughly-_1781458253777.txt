App is running clean. The implementation is complete. Here's exactly what was built and why it works:

Root cause: party_name is stored as a plain string (denormalized) in every related collection — ledger_entries, transactions, misc_charges, recurring_templates. The PartyDetailView filters all records with where('party_name', '==', party.name). When you edited a party's name, the party document updated but all the linked records still had the old name → the detail view showed zero records for the "new" party.

New service: src/services/partySync.ts

Runs after any party edit and updates 4 collections in parallel Firestore batches (chunked at 400 ops each to stay under Firestore's 500-op limit):

Collection	Fields updated
ledger_entries	party_name, address, site
transactions	party_name
misc_charges	party_name, party_id
recurring_templates	party_name
Misc charges use a JS-side filter (party_id === partyId || party_name === oldName) since there's no Firestore composite index — this catches records linked by either mechanism.

Hook in ManualEntryModal.tsx

After a successful party edit save, the cascade fires in the background (fire-and-forget after onClose()):

Save party doc  ──▶  UI closes immediately  ──▶  cascade runs in background
                                                      ↓
                                            "Updated N linked records" toast

Name change: all records re-point to the new name → PartyDetailView finds them
GST toggle (non-GST → GST party): the party doc gains a gstin field; cascade updates the address/site on historical ledger entries
Address/site change: historical invoices also get the new address (per user's requirement)
No change needed: if nothing denormalized changed, the cascade still runs but updates are minimal (Firestore deduplicates no-op field updates)
Never blocks UI: the toast "Updated N linked records" appears a few seconds after the modal closes, when the batch commits complete
