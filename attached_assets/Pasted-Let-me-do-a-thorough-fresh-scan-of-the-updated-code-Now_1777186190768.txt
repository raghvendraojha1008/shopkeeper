## 🔴 Critical Issues

**1. Back Button Listener Re-registers on Every State Change — Still Not Fixed**
```ts
}, [showManualModal, showCommandModal, activeTab]); // ← re-registers on every tab change
```
The previous Android version fixed this with `useRef`. This web version reverted to re-registering the Capacitor back button listener on every tab switch, modal open, and modal close. Each re-registration briefly gaps between `remove()` and `addListener()` — during which Android's hardware back button does nothing, and more critically, each old listener that wasn't cleanly removed before the async `addListener` resolves can fire **twice** for one back press, causing double-navigation.

**7. Settings Saved to `user.uid` — Not `dataUid` — Breaking Staff-Viewing-Admin Scenario**
```ts

```
In `SettingsView`'s `onUpdateSettings` handler in `App.tsx` (line 348), settings are saved to `user.uid` always. `LockScreen` locks the app once at startup (via `sessionStorage`) and never re-locks based on an idle timeout, despite `timeout` being a first-class field in `SecuritySettings`. **11. `onboardingKey` Stored in Both `localStorage` AND `sessionStorage` — Redundant and Inconsistent**
```ts
// On complete:
localStorage.setItem(onboardingKey, 'true');
sessionStorage.setItem(onboardingKey, 'true');

// On check:
const alreadyOnboarded =
  localStorage.getItem(onboardingKey) ||
  sessionStorage.getItem(onboardingKey);
```
Storing in both is redundant — `localStorage` already persists across sessions, making `sessionStorage` unnecessary. More importantly, if a user clears `localStorage` (app reinstall, data wipe) but not `sessionStorage` (tab still open), they won't see onboarding even though `localStorage` was cleared — which was the intended trigger. The `sessionStorage` fallback defeats the reinstall detection logic.

**14. `isInterstate` Calculation in `LedgerEntryDetailView` Silently Returns `false` When Party Has No GSTIN**
```ts
const isInterstate = (settings?.profile?.gstin || '').substring(0,2) !== 
                     (party?.gstin || '').substring(0,2) && !!party?.gstin;
```
When `party?.gstin` is empty, `!!party?.gstin` is `false`, so `isInterstate` is always `false` for unregistered parties. This is the correct behavior, but the issue is that `''.substring(0,2)` returns `''` for both seller and buyer when GSTIN is missing, and `'' !== ''` is `false` — so the logic accidentally produces the right answer for the wrong reason. This will silently break if either GSTIN field is `null` instead of `undefined` or `''`, since `null.substring` throws.

**15. `refreshKey` Pattern Forces Full Component Re-mount Instead of Data Refresh**
```ts
<DashboardView key={refreshKey} .../>
<InventoryView key={refreshKey} .../>
```
Incrementing `refreshKey` destroys and recreates the entire component tree for that view, losing all local state (scroll position, open accordions, search terms, etc.). For a web app this is especially jarring. Views should expose a `refresh()` method or use context invalidation — which `invalidateAll` from `DataContext` already provides — rather than full unmount/remount via `key`.
