🧠 AI Agent Prompt: Fix Party Edit Synchronization (Cascade + UI Consistency)
🎯 Goal
When a user edits a party name (or any other party field like address, GSTIN, site, etc.), the system must:

Update the party document in Firestore.

Cascade the change to all related records across collections (ledger_entries, transactions, misc_charges, recurring_templates) that reference that party, replacing the old party name (and other fields) with the new values.

Ensure no stale data remains: old party name must not appear in any record after the update.

Reflect the changes immediately in all UI views (Parties list, Party Detail, Ledger, Transactions, Pending, etc.) without duplicate entries or visual glitches.

🐛 Observed Issues
After editing a party name, the Parties list shows multiple entries for the same party (old name and new name), indicating a cache inconsistency.

Records (ledger, transactions) that belonged to the old party name are not updated with the new name — they still show the old name.

Newly created records after the edit are correctly associated with the new party name, but old records remain "orphaned" under the old name, causing fragmentation.

🔍 Root Cause Analysis (to be verified by AI)
Cascade function syncPartyToRecords is called asynchronously after the party document is saved, but the UI may refresh its data before the cascade completes, leading to stale views.

React Query cache invalidation (invalidateQueries for ledger and transactions) happens after the cascade (in partySync.ts), but the PartyDetailView and PartiesView may have already re-rendered with stale data.

Optimistic updates in PartiesView (handleLocalUpdate) update the party list correctly, but the ledger/transactions caches are not invalidated immediately after the cascade, so the detail view still shows old data.

The party_id field might be missing from some older records, causing the cascade to fall back to querying by party_name, but if the name is changed, those records are missed (though the code queries by both, but there could be a timing issue).

Duplicate entries in the parties list may arise from:

Optimistic update adding the new party without removing the old one (if ID mismatch).

Multiple re-renders causing stale cache entries.

The setPartiesCache merge logic not correctly identifying the existing party.

✅ Desired Behavior
Editing a party should be a transactional operation: either all related data is updated, or none.

Immediate UI feedback: the party list updates instantly (optimistic), and all related lists (ledger, transactions) refresh automatically as soon as the cascade finishes.

No duplicate parties: after the edit, only one party appears with the new name.

All historical records (ledger entries, transactions, misc charges) should reflect the new party name and other updated fields (address, GSTIN, site).
