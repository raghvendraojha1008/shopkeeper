**10. `parseID` Regex Fails for Multi-Character Prefixes Like `REC-` and `PAY-`**
```ts
const match = id.match(/^([A-Z]+-?)(\d+)$/);
```
The regex `[A-Z]+-?` matches one or more uppercase letters optionally followed by one `-`. For `'REC-101'`:
- `[A-Z]+` matches `REC`
- `-?` matches `-`
- Result: prefix `'REC-'`, number `101` ✅

But for `'PAY-101'`:
- `[A-Z]+` matches `PAY`
- `-?` matches `-`
- Result: prefix `'PAY-'`, number `101` ✅

This works, but for any ID that doesn't end in digits (e.g. a corrupted ID like `'S-101A'`), `parseID` returns `null` silently — which causes `seedCountersFromFirestore` to skip that record entirely, potentially under-seeding the counter and issuing a duplicate ID matching the skipped record.

```ts

  initCountersForUser(user.uid);
  // ...
}, [user, authLoading, dataUid]);
```
`dataUid` changes whenever the staff member switches viewed firms. Every firm switch re-calls `initCountersForUser(user.uid)` (note: `user.uid`, not `dataUid`) — but since it's guarded by `userId === _currentUserId`, it's a no-op. However the entire `initApp` async chain (including `AutoBackupService`, `DailySnapshotService`, `AutoReminderService`) re-runs on every `dataUid` change, meaning daily backup and snapshot checks fire every time a staff member switches which firm they're viewing — potentially triggering duplicate backups.
